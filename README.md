# 宜兴四人旅行静态站规划

这个仓库用于规划并制作一个发给朋友的宜兴旅行静态网站。当前阶段不是做最终页面，而是先明确网站的内核、交互方向、地点数据模型和后续需要讨论的地点选项。

## 项目目标

- 面向四位朋友的宜兴旅行，包括两男两女，其中包括我。
- 最终产物必须是可部署到 GitHub Pages 的纯静态网站。
- 页面气质应简约、现代、sans-serif，不做传统旅游攻略式堆叠。
- 核心表达不是“宜兴景点清单”，而是“通过泥、茶、竹、洞、水、夜生活进入宜兴”。
- 网站应支持朋友们按兴趣拼配行程，包括景点、体验、住宿等模块。

## 当前核心想法

宜兴可以被设计成一个“感官菜单”：

- 丁蜀 / 紫砂 / 窑火：红色、暖色、火光感。
- 春日 / 阳羡茶 / 田野：绿色、黄绿色、轻盈明亮。
- 竹 / 洞 / 水：绿色、蓝绿色、湿润、清凉。
- 夜晚 / 住宿 / 饭局：低饱和、暖光、松弛。

整个网站可以是一个页面，但根据鼠标 hover、选中状态或菜单切换，用 JavaScript 改变页面 CSS theme，使不同区域呈现不同感官氛围。

## 设计原则

- 单页优先，避免复杂路由。
- 静态优先，不依赖后端、数据库或登录。
- 内容体验像菜单，而不是景点列表。
- 视觉克制，不用过度装饰或旅游平台式卡片堆叠。
- 交互要轻：hover、select、drag、toggle 即可。
- 行程拼配要有实际用处：选择地点时能提示区域跨度和大致交通时间。
- 地点关系用网络图表达，只有确认存在直接交通的两点才连线，线长用交通时间近似表达。

## 下一步

在做 actual page 之前，需要先讨论并确认具体地点选项：

- 哪些地方进入候选池。
- 每个地点属于哪个感官主题。
- 每个地点的位置代码是什么。
- 哪些地点适合四人一起，哪些更适合可选模块。
- 住宿是否放在和景点同等级的可拖拽板块中。
- 哪些地点之间存在直接交通，哪些必须经过第三地或不宜连线。
- 交通时间是写死估算，还是先用区域间矩阵粗略计算。
